# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目概述

《渡世人》是一个赛博朋克/废土科幻 IP 项目，包含小说原作、电影剧本、30分钟动画改编方案，以及使用 AI 工具（Kling、Midjourney、Runway、Suno、ElevenLabs 等）进行概念验证的完整物料。视觉风格参考《攻壳机动队》《银翼杀手2049》《赛博朋克2077》。。

## 核心文件层级

### 原作与改编。
- **`渡世人-小说.md`** — 小说原版（正本）。任何故事改动的基准。
- **`渡世人-小说-v2.md`** / **`v3-李碧华风格.md`** / **`v4-三体风格.md`** — 风格实验版本。不是正本，用于探索不同叙事风格。
- **`渡世人-电影剧本.md`** — 电影剧本改编版。
- **`渡世人-电影剧本-抖音版.md`** — 竖屏短视频版剧本。
- **`剧本-30分钟.md`** — 30分钟动画剧本（当前动画项目的主要改编目标）。

### 设定与制作
- **`角色设定与分镜脚本.md`** — 角色外观、性格、视觉关键词、义眼特效规则、分镜初稿。修改角色设定时必须同步更新 `概念验证/风格圣经-每镜必查.md` 中对应的角色表格。
- **`动画制作方案.md`** — AI 辅助动画制作管线方案（5个阶段：剧本→视觉设计→音频→视频生成→后期），含工具推荐和成本估算。
- **`AI生成提示词库.md`** — 按场景组织的 AI 图像/视频提示词库（英文提示词为主，因为 Midjourney/Kling/Runway 对英文响应更好）。每个场景同时包含 Midjourney 和 Kling/Wan 两套提示词。

### 概念验证（`概念验证/`）
- **`风格圣经-每镜必查.md`** — AI 生成 consistency 检查表。生成任何角色图像/视频前必须对照此表，列出 AI 常犯错误。修改角色设定时需要同步更新此文件。
- **`最终镜头清单.md`** — 5分钟概念验证的17个镜头清单，含进度追踪。
- **`螺丝与灰-纠葛线.md`** — 角色关系线文档。
- **`世界观深度设定.md`** / **`世界观与人设大全.md`** — 世界设定详情。
- **`提示词生成器.md`** — 提示词生成辅助工具说明。
- **`旁白音频/`** — TTS 生成的旁白音频（.aiff 和 .wav 格式），11段。
- **`生成旁白.sh`** / **`生成旁白-男声.sh`** — 旁白生成脚本。

### 素材文件
- **`灰定妆照1.png`** / **`灰定妆照2.png`** / **`定妆照3_0.png`** — 主角灰的角色定妆照，用作 AI 图生视频的参考图。

- **`hailuo-assets/`** / **`hailuo-assets.zip`** — Hailuo AI 生成的视频素材（10个 .mp4）。

## 工作约定

1. **改动故事/角色设定时，必须检查是否需要同步更新**：`风格圣经-每镜必查.md`、`角色设定与分镜脚本.md`、`AI生成提示词库.md` 中对应的描述。
2. **生成任何 AI 图像/视频提示词前，对照 `风格圣经-每镜必查.md`** 检查角色特征是否正确。
3. **提示词使用英文**（Midjourney/Kling/Runway 等工具对英文响应更好），中文仅用于场景说明。
4. **素材文件（.mp4, .png, .aiff, .wav）是生成产物**，修改应通过调整提示词或参数重新生成，而非直接编辑二进制文件。
5. **`/88` 提交流程（三步，不可跳过任何一步）**：
   - **第一步**：`git diff --cached` 查看暂存区所有改动。
   - **第二步**：向用户分点概括本次改动——列出每个文件、具体改了什么、为什么改。不能只说"更新了某某文件"。必须让用户不看 diff 也能理解改了什么。如果提交信息有附加说明，直接用；否则根据概括自动生成简洁中文提交信息。
   - **第三步**：`git commit` + `git push`。若推送被拒，先 `git pull --rebase`；rebase 有冲突则停下告知用户。
   - **禁止**跳过总结直接 commit+push。即使用户说"直接提交"，也必须先简要概括改动再提交。
6. **`/77` 拉取后总结（三步，不可跳过任何一步）**：
   - **第一步**：`git pull`。
   - **第二步**：`git log` 查看新增的提交信息，结合 `git diff` 查看改动文件内容。
   - **第三步**：向用户总结拉取内容——提炼核心变化、每个改动"改了什么、为什么改、跟其他设定的关联"。不能只列文件名和行数。篇幅适中，一两句话概括每个重要改动，不展开大段原文。
